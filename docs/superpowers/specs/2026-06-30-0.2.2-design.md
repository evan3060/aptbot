## 1. 背景与目标

### 1.1 版本定位

0.2.2 是 0.2.x 系列的延续，目标是把 aptbot 从"可用"演进为"可靠 + 可扩展 + 体验流畅"，并为 0.3.0 多 agent 系统建立扩展性基础。

原 PLAN-L2 的 11 项历史功能降级为 0.2.2（版本号对齐：0.3.0 留给多 agent 系统建立）。

### 1.2 核心目标

- **可靠性**：偿还 0.2.x 遗留技术债（ring buffer 内存、turn_busy 反馈、JSONL 历史持久化、HttpOnly cookie）
- **扩展性**：Provider 故障转移 / Config 热重载 / Hook 系统 / Skills 系统 / Channel 抽象
- **体验**：Session 自动摘要 / /session 动态属性 / L1 索引 Skill

### 1.3 为 0.3.0 铺路

0.2.2 的以下 task 是 0.3.0 多 agent 系统的前置依赖：
- Task 5 MixinProvider → 0.3.0 不同 agent 用不同 provider
- Task 7 Hook 系统 → 0.3.0 注入 agent 个性
- Task 8 Skills 系统 → 0.3.0 agent 独立技能绑定
- Task 11 /session 动态属性 → 0.3.0 agent 级配置
- Task 12 Channel 抽象 → 0.3.0 多 agent UI 路由

### 3.3 0.2.x 技术债设计契约

**Task 1 ring buffer 分片**：
- 单 sessionKey 上限 1000 不变
- 全局 50000 上限触发 LRU 淘汰最旧 sessionKey 的全部 buffer
- sessionKey refCount 归零时清理

**Task 2 turn_busy**：
- 同 sessionKey 已有 turn 执行时，新消息入队前发送 `{ type: 'turn_busy', position: N }`
- turn 完成后不发送 turn_ready（前端靠 turn_end 事件自然恢复）
- 前端监听 turn_busy 显示"等待中... (前方 N 条消息)"

**Task 3 JSONL 历史持久化**：
- ring buffer 为空（服务器重启后）时，调用 `readHistoryForReplay(id)` 读 JSONL
- 仅返回 type === 'message'，不返回 tool_call（避免泄漏内部状态）
- 标记 `replay: true`，前端不重复渲染
- **关键约束**：agent 仍受 `data/sessions/` 访问禁令，此路径仅 wsServer 使用

**Task 4 HttpOnly cookie**：
- POST /api/register /api/login 成功时设置 Set-Cookie
- Cookie: `aptbot_token=<token>; HttpOnly; Secure; SameSite=Strict; Path=/; Max-Age=2592000`
- GET /api/me 优先读 cookie，其次 Authorization: Bearer
- WebSocket 连接 token 来源优先级：URL ?token= > cookie > sessionStorage
- 前端 fetch 自动带 cookie（`credentials: 'include'`）

**Task 10 Session 自动摘要**：
- turn_end 后检查 session 是否已有 label
- 无 label 时异步调用 LLM 生成摘要（≤20 字符）
- 摘要 prompt: `"Summarize this conversation in ≤20 chars. No punctuation. No quotes."`
- 用户手动 `/label` 后跳过自动摘要
- LLM 失败时不报错，保留默认 label

## 5. 错误处理与边界

### 5.1 各 task 边界

| Task | 边界场景 | 处理 |
|---|---|---|
| 1 | 全局 50000 触发时无 sessionKey 可淘汰 | 丢弃最旧 buffer + warn |
| 2 | turn_busy 发送失败 | 静默忽略（不阻塞主流程） |
| 3 | JSONL 文件损坏 | 增量流式解析 + 自动截断修复 |
| 4 | cookie 被禁用 | fallback 到 sessionStorage |
| 5 | 所有 provider 失败 | 抛 AggregateError |
| 5 | 流式已 yield 后出错 | 不切 provider，yield 错误块 |
| 6 | 配置文件损坏 | 保留旧配置 + channel 错误通知 |
| 7 | hook 抛错 | 吞掉 + stderr 打印 + 不影响主流程 |
| 8 | SKILL.md frontmatter 损坏 | 返回 SkillDiagnostic warning + 跳过该 skill |
| 9 | L1 索引超 4K token | 截断 + 仅注入 lastUsed 前 N 个 + 全部名字列表 |
| 10 | LLM 摘要失败 | 不报错，保留默认 label |
| 11 | 非法属性值 | 返回错误提示 + 列出合法值 |
| 12 | Channel 接口实现错误 | 抛错 + 不影响其他 channel |

### 5.2 性能边界

- **MixinProvider 串联延迟**：仅在 retryable 错误后 fallback，fatal 立即抛出
- **Hook 系统性能**：hook 必须同步且不阻塞主流程（fire-and-forget 异步需求）
- **JSONL 历史读取**：限制 limit=20，仅返回最近 N 条
- **L1 索引 token 预算**：4K token 上限，超出截断

### 5.3 熔断机制（研发流程级）

- 遇到 3 次连续不可修复的测试失败：触发熔断，停止当前 task
- 打印错误栈，标记 task 为 failed，记录依赖
- 切换到其他无依赖 task，全部完成后再回来修复

## 7. 风险与依赖

### 7.1 内部依赖

- Task 9 依赖 Task 8（Skills 系统基础）
- Task 11 依赖 Task 5（MixinProvider 广播机制）
- Task 13 依赖 Task 1-12 全部完成
- Task 14 依赖 Task 13
- Task 15 依赖 Task 14

### 7.2 外部依赖

无新增第三方依赖（hook 系统 + Skills 系统用纯 TS 实现）。

### 7.3 风险点

1. **MixinProvider 串联延迟**：多 provider fallback 时累计延迟。仅在 retryable 错误后 fallback。
2. **Config 热重载竞态**：修改配置文件时可能有部分写入。用原子写（write-to-tmp + rename）。
3. **Hook 系统性能**：8 个 hook 点每个 turn 调用多次。hook 必须同步且不阻塞。
4. **JSONL 历史读取性能**：大 session（10000+ 消息）读取慢。限制 limit=20。
5. **Skills 系统与 0.3.0 多 agent 的衔接**：0.2.2 建立 Skills 系统基础，0.3.0 在此之上实现 agent 独立技能绑定。需保证接口设计前瞻性。
6. **/session 动态属性与 MixinProvider 的耦合**：/session 依赖 MixinProvider 广播机制，需保证 MixinProvider 先完成。
7. **文档同步遗漏**：0.2.x 封仓时发现各类文档未更新。0.2.2 研发流程新增 A11-A14 文档同步步骤 + B4-B8 封仓文档步骤。
