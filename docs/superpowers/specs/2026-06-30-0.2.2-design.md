## 1. 背景与目标

### 1.1 版本定位

0.2.2 是 0.2.x 系列的延续，主题是"可靠性 + 扩展性 + 体验"。原 PLAN-L2 的 11 项历史功能降级为 0.2.2（版本号对齐：0.3.0 留给多 agent 系统建立）。

### 1.2 核心目标

- **可靠性**：偿还 0.2.x 遗留技术债（ring buffer 内存、turn_busy 反馈、JSONL 历史持久化、HttpOnly cookie）
- **扩展性**：Provider 故障转移 / Config 热重载 / Hook 系统 / Skills 系统 / Channel 抽象
- **体验**：Session 自动摘要 / /session 动态属性 / L1 索引 Skill

### 1.3 为 0.3.0 铺路

0.2.2 的以下 task 是 0.3.0 多 agent 系统的前置依赖：
- Task 5 MixinProvider → 0.3.0 不同 agent 用不同 provider
- Task 7 Hook 系统 → 0.3.0 注入 agent 个性
- Task 8 Skills 系统 → 0.3.0 agent 独立技能绑定
- Task 11 /session 动态属性 → 0.3.0 agent 级配置
- Task 12 Channel 抽象 → 0.3.0 多 agent UI 路由

## 4. 技术方案与边界

### 4.1 Task 1 — per-sessionKey ring buffer 分片 + LRU

**技术方案：**
- 单 sessionKey 上限保持 1000（不变）
- 新增全局总条目上限 50000，触发时按 LRU 淘汰最旧 sessionKey 的全部 buffer
- 维护 sessionKey → refCount 映射，refCount 归零时清理对应 buffer
- LRU 排序依据：最近一次写入或读取时间

**边界条件：**
- 全局 50000 触发时若无可淘汰 sessionKey（罕见，所有 sessionKey 都在活跃使用）→ 丢弃最旧 buffer + warn 日志
- sessionKey refCount 归零判定：最后一个 channel 断开后归零
- LRU 淘汰后新 sessionKey 写入不受影响

**约束：**
- 不改变现有 ring buffer 单 sessionKey 1000 上限行为
- LRU 淘汰必须整 sessionKey 淘汰，不能跨 sessionKey 部分淘汰
- 内存释放后不持有引用，便于 GC

**安全控制：**
- refCount 归零清理必须原子，避免并发 channel 断开导致重复清理
- LRU 淘汰时记录 warn 日志，便于监控内存压力

### 4.2 Task 2 — turn_busy 响应

**技术方案：**
- 在 `runInboundLoop` 串行化逻辑中，同 sessionKey 已有 turn 执行时，新消息入队前发送 turn_busy 消息
- 消息格式：`{ type: 'turn_busy', position: N }`，N 为前方排队消息数
- turn 完成后不主动发 turn_ready，前端靠 turn_end 事件自然恢复
- 前端监听 turn_busy 显示"等待中... (前方 N 条消息)"，收到 turn_start/turn_end 清除

**边界条件：**
- turn_busy 发送失败时静默忽略（WebSocket 已断开等场景），不阻塞主流程
- 不同 sessionKey 互不影响（A sessionKey 排队不影响 B sessionKey）
- position 计数包含当前消息本身

**约束：**
- turn_busy 仅在"同 sessionKey 已有 turn 执行"时发送，首条消息不触发
- 不引入 turn_ready 事件（避免与 turn_end 重复）
- 前端清除等待提示的时机与 turn_start 对齐

**安全控制：**
- turn_busy 不泄漏其他 sessionKey 信息
- position 仅反映当前 sessionKey 队列深度

### 4.3 Task 3 — JSONL 历史持久化

**技术方案：**
- ring buffer 未命中时（服务重启后清空），调用 `readHistoryForReplay(id, limit)` 读 JSONL
- 仅返回 type === 'message'，不返回 tool_call（避免泄漏内部状态）
- 返回消息标记 `replay: true`，前端不重复渲染（不触发 appendUserMsg 等副作用）
- limit 默认 20，仅返回最近 N 条

**边界条件：**
- JSONL 文件不存在时返回空数组（新 session）
- JSONL 文件损坏时增量流式解析 + 自动截断修复（fs.truncateSync 截断到损坏点）
- ring buffer 命中时不读 JSONL（性能优先）

**约束：**
- **关键安全约束：** agent 仍受 `data/sessions/` 访问禁令，此路径仅 wsServer 使用
- readHistoryForReplay 不暴露给 agent 工具
- 回放消息不触发系统 prompt 重建

**安全控制：**
- readHistoryForReplay 仅限 wsServer 调用，不进入 agent 工具表
- 回放消息过滤 tool_call 防止内部状态泄漏
- 文件损坏修复时保留未损坏部分，不删除整个文件

### 4.4 Task 4 — HttpOnly cookie 安全增强

**技术方案：**
- POST /api/register /api/login 成功时设置 Set-Cookie 响应头
- Cookie 属性：`HttpOnly; Secure; SameSite=Strict; Path=/; Max-Age=2592000`（30 天）
- GET /api/me 优先读 cookie，其次读 Authorization: Bearer header
- WebSocket 连接 token 优先级：URL ?token= > cookie > sessionStorage
- 前端 fetch 自动带 cookie（`credentials: 'include'`）

**边界条件：**
- cookie 被禁用（浏览器设置）时 fallback 到 sessionStorage，不报错
- HTTPS 环境才设置 Secure 属性（本地 HTTP 开发环境降级）
- SameSite=Strict 阻止跨站请求携带 cookie

**约束：**
- token 同时存 cookie + sessionStorage（双写），优先读 cookie
- WebSocket URL ?token= 优先级最高（支持跨设备链接）
- cookie 过期后自动重新登录

**安全控制：**
- HttpOnly 防止 XSS 读取 token
- Secure 仅 HTTPS 传输
- SameSite=Strict 防 CSRF
- Max-Age=2592000（30 天）平衡安全与体验
- 不存储敏感信息到 cookie（仅 token）

### 4.5 Task 5 — MixinProvider 多 provider 故障转移

**技术方案：**
- 多 provider 按 priority 升序串联（priority 小的优先）
- 前 provider 失败（fatal 除外）自动 fallback 到下一个 provider
- retryable 错误（429/5xx）重试 3 次后才 fallback
- fatal 错误（401/403/400）立即抛出，不 fallback
- 流式已 yield 后出错不切 provider（避免重复输出），直接 yield 错误块
- springBackMs 后弹回主 provider（避免长期使用备 provider）
- 广播属性到所有子 provider（temperature/maxTokens 等）
- 所有 provider 失败抛 AggregateError

**边界条件：**
- 单 provider 成功时不触发 fallback
- 流式首字节前出错：正常 fallback
- 流式首字节后出错：不 fallback，yield 错误块
- 所有 provider 同时失败：AggregateError 包含全部错误

**约束：**
- 同协议约束：所有子 provider 必须支持相同协议（如都支持 streaming）
- 广播属性同步到所有子 provider（非惰性）
- springBackMs 计时从 fallback 时刻开始

**安全控制：**
- API key 不泄漏到日志
- AggregateError 不暴露内部 provider 配置
- fallback 决策可观测（日志记录切换原因 + 目标 provider）

### 4.6 Task 6 — Config 热重载

**技术方案：**
- 监听 `config/aptbot.json` 的 mtimeNs 变化（懒加载，非 fs.watch）
- AgentSession 在 beforeTurn 检查 mtimeNs，变化时整体重载
- 当前 turn 用旧配置快照，下个 turn 用新配置
- 校验失败降级到旧配置 + channel 错误通知
- stop() 清理资源

**边界条件：**
- 配置文件不存在时保留默认配置
- 配置文件部分写入（写过程中读取）→ 校验失败降级
- 并发读：mtimeNs 检查是原子的

**约束：**
- 懒加载策略：不主动 fs.watch，仅在 beforeTurn 时 stat 文件
- 当前 turn 不受配置变化影响（快照机制）
- 校验失败必须降级，不抛错中断服务

**安全控制：**
- 配置文件权限检查（仅 owner 可读写）
- 热重载不触发服务重启
- 错误通知走 channel 事件，不暴露内部错误细节给用户

### 4.7 Task 7 — Hook 系统（8 hook 点）

**技术方案：**
- 8 个 hook 点：agent_before / agent_after / turn_before / turn_after / llm_before / llm_after / tool_before / tool_after
- 同步执行（不异步等待）
- ctx 允许 mutate（链式传递，后续 hook 看到前一个的修改）
- priority 升序排序（小的先执行）
- 两层插件目录：`~/.aptbot/hooks/`（workspace）+ `.agents/hooks/`（builtin），workspace 覆盖 builtin 同名
- 无沙箱（hook 直接访问 Node.js API）
- hook 抛错吞掉 + stderr 打印 + 不影响主流程

**边界条件：**
- hook 注册空函数：不报错，正常执行
- hook 执行超时：无超时限制（同步，开发者责任）
- 多个 hook 同 priority：按注册顺序执行
- ctx 修改不可逆（链式传递）

**约束：**
- 同步执行，不阻塞主流程（hook 必须快速返回）
- 无沙箱：hook 拥有完全权限，由开发者保证安全
- 两层目录加载顺序：builtin 先加载，workspace 后加载覆盖

**安全控制：**
- hook 抛错不影响主流程（吞异常 + stderr）
- hook 不修改主流程返回值（仅修改 ctx）
- workspace hook 仅 owner 可写（权限检查）

### 4.8 Task 8 — Skills 系统基础（§8.5/§8.6）

**技术方案：**
- 两层加载：workspace `~/.aptbot/skills/` + builtin `src/skills/`
- workspace 覆盖 builtin 同名 skill
- 最小 frontmatter：name / description / disableModelInvocation
- 校验：name（a-z0-9-, ≤64 字符）+ description（≤1024 字符）
- 解析失败返回 SkillDiagnostic warning + 跳过该 skill
- 全量 name+description 注入 system prompt
- ExecutionEnv 抽象：cwd / env vars / permissions

**边界条件：**
- skill 目录不存在时返回空列表（不报错）
- frontmatter 损坏时返回 SkillDiagnostic warning + 跳过
- 同名 skill 覆盖时记录 info 日志

**约束：**
- 最小 frontmatter：不强制要求其他字段
- ExecutionEnv 不暴露给 skill 实现者（仅框架内部使用）
- disableModelInvocation=true 时 skill 不注入 system prompt

**安全控制：**
- skill 文件权限检查（仅 owner 可读写）
- ExecutionEnv 限制 cwd 在 skill 目录内
- skill 加载不执行任何代码（仅解析 frontmatter）

### 4.9 Task 9 — L1 索引 Skill

**技术方案：**
- 扩展 Skill 接口：contentLines / contentBytes / tags / lastUsed
- formatSkillsForSystemPrompt 按 lastUsed 降序排序
- 总 token 超 4K 预算时截断，仅注入 lastUsed 前 N 个 + 全部名字列表
- read_file 读取 skill 文件时特判更新 lastUsed
- 热重载联动：Config 热重载时 Skills 也重载

**边界条件：**
- 0 个 skill 时返回空字符串
- 单 skill 超 4K 预算时仅注入该 skill 的名字 + description
- lastUsed 未初始化时按 0 处理（排最后）

**约束：**
- 4K token 预算硬上限，超出必须截断
- lastUsed 持久化到文件（跨重启保留）
- read_file 特判仅对 skill 文件生效

**安全控制：**
- lastUsed 文件权限（仅 owner 可读写）
- 截断时保留名字列表（agent 至少知道有哪些 skill）

### 4.10 Task 10 — Session 自动摘要命名

**技术方案：**
- turn_end 后检查 session 是否已有 label
- 无 label 时异步调用 LLM 生成摘要（≤20 字符）
- 摘要 prompt 固定："Summarize this conversation in ≤20 chars. No punctuation. No quotes."
- 用户手动 /label 后跳过自动摘要
- LLM 失败不报错，保留默认 label
- 新增 hasCustomLabel() 方法

**边界条件：**
- LLM 调用超时：不报错，保留默认 label
- 摘要超 20 字符：截断到 20 字符
- session 首条消息触发摘要（无 turn_end 也尝试）

**约束：**
- 异步生成不阻塞主流程
- 用户手动 /label 优先级最高（永久跳过自动摘要）
- 摘要 prompt 不可配置（固定）

**安全控制：**
- 摘要不包含敏感信息（LLM 调用前过滤）
- 摘要失败不暴露 LLM 错误给用户

### 4.11 Task 11 — /session 动态属性

**技术方案：**
- 白名单 5 项：temperature / maxTokens / reasoningEffort / thinkingType / thinkingBudgetTokens
- 文件值逃生口：非白名单项写入文件供 agent 读取
- JSON 自动解析：number / boolean / null
- 内存态存储（不持久化，重启后重置）
- /session.reset 重置所有属性到默认值
- MixinProvider 广播属性到子 provider
- 非法属性值返回错误 + 列出合法值

**边界条件：**
- 属性值类型不匹配（如 temperature 传字符串）：返回错误
- /session 无参数：列出所有属性当前值
- /session <attr> 无值：返回当前值

**约束：**
- 白名单硬编码，不支持动态扩展
- 文件值逃生口仅写文件，不进入内存态
- MixinProvider 广播同步到所有子 provider

**安全控制：**
- 非法属性名拒绝（防注入）
- 属性值范围校验（temperature 0-2 等）
- /session.reset 不影响其他 session

### 4.12 Task 12 — Channel 接口抽象

**技术方案：**
- 方案 E 类型化 bus + AgentEventEnvelope
- Channel 接口：type / send / close / isAlive
- bindSession(sessionKey, channel) 多对一共享
- IM channel 管理 sessionKey 映射无需 ?session= 参数
- WebSocket 仍作为 Channel 实现工作

**边界条件：**
- channel 断开时自动 unbind
- 多 channel 共享同一 sessionKey：消息广播到所有 channel
- channel type 未知时拒绝注册

**约束：**
- Channel 接口最小化（4 个方法）
- AgentEventEnvelope 类型化（type 字段）
- bindSession 多对一（多个 channel 可绑同一 sessionKey）

**安全控制：**
- channel 注册需认证（token）
- IM channel 的 sessionKey 映射持久化（跨重启）
- channel 断开不泄漏 sessionKey 信息

### 7.2 外部依赖

无新增第三方依赖（hook 系统 + Skills 系统用纯 TS 实现）。

### 7.3 风险点

1. **MixinProvider 串联延迟**：多 provider fallback 时累计延迟。仅在 retryable 错误后 fallback。
2. **Config 热重载竞态**：修改配置文件时可能有部分写入。用原子写（write-to-tmp + rename）。
3. **Hook 系统性能**：8 个 hook 点每个 turn 调用多次。hook 必须同步且不阻塞。
4. **JSONL 历史读取性能**：大 session（10000+ 消息）读取慢。限制 limit=20。
5. **Skills 系统与 0.3.0 多 agent 的衔接**：0.2.2 建立 Skills 系统基础，0.3.0 在此之上实现 agent 独立技能绑定。需保证接口设计前瞻性。
6. **/session 动态属性与 MixinProvider 的耦合**：/session 依赖 MixinProvider 广播机制，需保证 MixinProvider 先完成。
7. **文档同步遗漏**：0.2.x 封仓时发现各类文档未更新。0.2.2 研发流程新增 A11-A14 文档同步步骤 + B4-B8 封仓文档步骤。
