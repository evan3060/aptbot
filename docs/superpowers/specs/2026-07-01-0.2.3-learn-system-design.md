# aptbot 0.2.3 学习系统设计：知识栏目 + 反馈区 + 19 篇文章

**日期：** 2026-07-01
**版本：** v0.2.3
**状态：** 待用户审核

## 目标

aptbot 定位为"个人学习项目"——既是 agent 工具，也是 agent 学习教材。0.2.3 在网站首页新增知识栏目，展示 19 篇结构化文章（Track 1: Agent 体系实践 13 篇 + Track 2: AI 辅助编码实践 6 篇），覆盖 agent 原理、实现、演进路线，以及 AI 辅助编码的通用方法论。同步增加用户反馈区，访问者可提交想法/需求，记录到待办供项目维护者查阅。

文章以概念性理解为主（每篇 1500-2500 字），必要时辅助少量伪代码（≤20 行），概念清晰则不出现代码。读者画像为小白用户从 0 到概念性理解 agent。Track 2 与具体项目无关，是 AI 辅助编码的通用经验总结。

## 文章元数据 schema 与 Track 扩展性

### frontmatter schema

每篇 markdown 文件顶部 YAML 块，字段：

- slug：URL 路径标识，正则 ^[a-z0-9-]+$，max 64 字符，全局唯一
- title：显示标题，min 1 max 120
- description：卡片摘要，min 1 max 200
- track：Track 标识，必须在 TRACKS 注册表中
- chapter：Track 内章节分组名，min 1 max 32
- order：全局排序，int min 1 max 999，全局唯一
- difficulty：beginner | intermediate | advanced
- estimatedReadingTime：分钟数，int min 1 max 300
- status：published | planned
- prerequisites：前置文章 slug 列表，默认空数组
- lastUpdated：ISO 日期 YYYY-MM-DD
- tags：标签列表，默认空数组

校验失败行为：stderr warning（含文件名 + zod 错误详情）+ 跳过该文件，不阻塞启动。

### Track 注册表

在 src/learn/article-types.ts 声明 TRACKS 常量数组，每项含 id / title / description / order。0.2.3 含两项：agent-practice（Agent 体系实践）+ ai-coding-practice（AI 辅助编码实践）。未来扩展 Track 3 只需在数组追加一项 + 加对应 markdown 文件。

### ArticleState 接口

不可变快照，含 articles 数组 + tracks 数组 + bySlug Map + byTrack Map。ArticleLoader.getState() 返回当前快照。

## 19 篇文章完整大纲

### Track 1: Agent 体系实践（13 篇，编号 01-13）

围绕 aptbot 这个 agent 项目展开，从原理到实现到演进路线。

#### 入门篇（2 篇）

**01. Agent 是什么：从 chatbot 到 agent 的演进**
- chatbot / assistant / agent 三者差异（被动响应 vs 主动行动）
- ReAct 循环原理：Reasoning + Acting 交替，工具调用作为"行动"
- agent 核心组件心智模型：LLM 大脑 + 工具双手 + 记忆长尾 + 流式嘴巴
- 与传统软件的本质区别：声明式指令 → 自主决策
- 为什么 agent 重要：从"问什么答什么"到"交办任务完成"

**02. aptbot 全景：分层架构与设计哲学**
- 四层架构（access / bus / core / infrastructure）+ shared 跨层
- 严格单向依赖的意义（core 不感知 access 存在）
- 与 pi-agent / nanobot / GenericAgent 的对比定位
- aptbot 选 TypeScript 的理由
- 设计哲学："不是框架，不是 SaaS，而是'你的'agent"

#### 核心特性深入篇（8 篇）

**03. Provider 系统：多协议、多模型、故障转移**
- Api（协议）+ Provider（服务商）分离的设计
- 三种 API 协议：openai-completions / openai-responses / anthropic-messages
- TTFB 5s + 块间 1.5s 双时钟流式控制器
- 错误分类重试：401/403/400 fatal；429/5xx 指数退避
- MixinProvider 多 provider 故障转移 + springBackMs 弹回
- 流式已 yield 后出错不切 provider
- 与 nanobot FallbackProvider / GA MixinSession 对比

**04. Tool 系统：声明式 registry 与安全边界**
- AgentTool 接口 + ToolRegistry 声明式注册
- 4 个内置工具：bash / read / edit / update_working_memory
- 参数 TypeBox schema 校验
- 30s 硬超时 + 大文件 OOM 防护
- 路径遍历防护（path-guard）
- systemPrompt 安全约束
- 与 GA code_run 万能工具的权衡

**05. Memory 系统：持久化、压缩、跨会话记忆**
- JSONL append-only 持久化设计
- 增量流式解析 + 破损行容错 + fs.truncateSync 自动修复
- SessionEntry 联合类型 + UUID 路径校验
- Working memory + /continue 跨 session 继承
- Compaction：80% 触发、30% 目标、token 三级估算
- 三层记忆架构规划（参考 GA L1/L2/L3）
- future: working dict（LLM 主动管理）

**06. Skills 系统：两层加载与自演化规划**
- 两层加载：workspace 覆盖 builtin
- 最小 frontmatter（name / description）
- 全量 description 注入 system prompt 的 token 成本
- L1 索引（lastUsed 降序 + 4K token 预算截断）
- read_file 特判更新 lastUsed
- 热重载联动
- future: 自演化 skill（参考 GA）

**07. Hook 系统：8 扩展点的插件机制**
- 8 hook 点拓扑：agent_before/after · turn_before/after · llm_before/after · tool_before/after
- 同步执行 + ctx mutate 链式传递 + priority 升序
- 两层插件目录（workspace 覆盖 builtin）
- 无沙箱设计哲学（hook 抛错吞掉 + stderr + 不影响主流程）
- 与 nanobot AgentHook / GA 8 hook 点对比
- 适用场景：日志 / 监控 / 审计 / 自动注入 steering

**08. Channel 与多端接入：TransportChannel 抽象**
- 方案 E 类型化 bus + AgentEventEnvelope 设计动机
- Channel 接口最小化（type / send / close / isAlive）
- wrapTransportChannel 适配器
- bindSession(sessionKey, channel) 多对一共享
- WebSocket 作为 Channel 实现
- dead-channel 自动 unbind
- future: IM channel 集成（Telegram 首通道）
- 与 nanobot 20+ IM channel / GA 4 IM app 对比

**09. Session 与多用户：持久化、隔离、多端同步**
- JSONL session 存储 + .meta.json sidecar
- UUID sessionId 路径校验
- claimSession 严格 ownership + forceClaimSession 共享转移
- per-sessionKey ring buffer（1000）+ 全局 LRU（50000）
- 历史回放（ring buffer 未命中 → JSONL 兜底）
- presence 广播
- session_changed 控制消息 + WebSocket 重连
- 多用户隔离：UserStorage + scrypt + Bearer token
- /sessions · /resume · /label · /session 动态属性命令

**10. 安全模型：多层防护与信任边界**
- systemPrompt 安全约束
- 30s 工具硬超时 + SIGTERM→SIGKILL
- TTFB / 块间双时钟流式控制
- 大文件 OOM 防护 + 工具结果截断
- 路径遍历防护
- JSONL 损坏自动修复
- HttpOnly + Secure + SameSite=Strict cookie
- WS token 三级优先级
- session ownership 跨用户 403
- API key 仅通过 .env
- X-Content-Type-Options + Cache-Control
- future: workspace 限制 + 权限模型（IM 接入后）

#### 可靠性 + UX 篇（1 篇）

**11. 错误处理与流式 UX：分层重试 + EventStream + reducer**

- 三层重试：传输 + 业务 + 语义
- 错误不持久化原则（防"400 poisoning"）
- AgentEvent 联合类型
- EventStream → UI reducer 模式（CLI Ink + WebUI Lit 共享 coreReducer）
- 流式渲染、回合中断、多端同步是事件流的自然消费
- resync 协议
- turn_busy 队列反馈
- SessionRef 可变引用（运行中切 session 不重启 loop）

### Track 2: AI 辅助编码实践（6 篇，编号 01-06）

与具体项目无关，是 AI 辅助开发的经验总结与学习方法论。案例可能提及 track1 项目中的问题，但仅作示例不深入细节。

**01. 开发流程与约束：superpower 工作流**
- superpower 标准流程：brainstorming → spec → writing-plans → executing-plans
- 文档分层约定：plan（描述性）/ design（技术方案）/ dev-workflow / design-notes
- plan 不含代码原则（描述性 + TDD 命令描述，代码在 TDD 阶段驱动）
- spec self-review + user review gate 双重审核
- subagent-driven-development 自动推进任务链
- TDD 红→绿循环（先见证 RED 错误，再最小代码修复 GREEN）
- 熔断机制（3 次连续失败停止 + 记录 + 跳过 + 回顾）
- 看板管理（subtask 完成后 [x] 标记 + 无缝推进）

**02. 编码准确性：TDD + 版本控制 + UAT 流程**
- TDD 在 AI 辅助开发中的实际应用
- 严禁跳过测试直接写业务代码的红线
- 版本号约定（Semantic Versioning + Keep a Changelog）
- UAT 核验清单流程（local / VPS / 新功能逐项 / 老功能回归四类核验）
- finishing-a-development-branch skill 的封仓收尾
- CHANGELOG / README / package.json / git tag 同步
- E2E 测试设计原则（覆盖 happy + error path，避免 zero expect）
- 测试基线维护（每次迭代不退化）

**03. spec 文档管理：从 brainstorming 到归档的全生命周期**
- spec 文档的价值：为什么 AI 辅助开发需要文档化的 spec
- spec 与 plan 与 design 的边界划分（描述性 / 技术方案 / 工作流）
- spec 文件组织结构（docs/superpowers/specs/YYYY-MM-DD-<topic>-design.md 模式）
- spec 的生命周期：brainstorming 产物 → user review → 实施依据 → 归档
- spec self-review 检查清单（placeholder / 一致性 / 范围 / 歧义）
- spec 与代码的同步：spec 变更如何追踪、过时 spec 如何处理
- 多版本 spec 的归档与回溯（git history + specs/ 目录）
- spec 作为团队协作媒介（PR review 依据、新人 onboarding 文档）
- 何时该写 spec、何时不必写（trivial 改动 vs 系统性变更的判断）

**04. 长期迭代维护实践：架构演进与可持续性**
- 版本规划：L1/L2/L3 路线图 vs 单版本 spec 的关系
- 迭代节奏：封仓 → UAT → 下一版本的循环
- 长期维护的测试基线管理（不退化红线 + 历史遗留 flaky 处理）
- 架构演进的"加法而不减法"原则（v0.1 → v0.2 抽象层叠加）
- design-notes 跨版本设计笔记的价值（约束 / 原则 / 决策回溯）
- project_memory 的维护（hard constraints / lessons learned）
- 回归测试与 E2E 测试在长期迭代中的作用
- 重构时机判断：何时该重构、何时该保持现状
- 依赖升级策略（marked / gray-matter 等新增依赖的评估）
- 文档同步：CHANGELOG / README / ARCHITECTURE 的版本同步
- 项目可持续性：避免技术债堆积、保持代码可读性、新人上手成本

**05. 边界与问题：能力边界、何时人工介入、踩坑**
- AI 擅长：模式化代码、测试用例、文档、重构、TDD 执行
- AI 不擅长：跨版本架构决策、需求歧义、视觉审美、性能微调
- 何时必须人工介入：架构方向、用户偏好、安全边界、商业判断
- 踩坑案例（通用，不绑定具体项目）：
  - AI 试图跳过测试直接写代码（必须用 skill 强制 TDD）
  - AI 在 plan 里写实现代码（必须用文档分层约束）
  - AI 重复尝试失败方案（必须用熔断机制止损）
  - AI 误读用户意图（必须用 AskUserQuestion 主动澄清）
  - AI 过度工程化（必须用 YAGNI 约束）
- AI 辅助开发的"约束即自由"哲学

**06. 方法与持续改进：协作、学习、知识沉淀**
- 如何与 AI 建立有效协作（提示词、上下文、约束规则）
- 如何从 AI 输出中学习（不只是接受结果，理解原理）
- 如何建立个人/项目的 AI 协作知识库（memory / project_memory）
- 如何评估 AI 输出质量（测试、review、对比）
- 如何持续优化协作流程（复盘、调整约束、迭代工作流）

## 反馈区 API 与存储

### API 契约

POST /api/feedback — 提交反馈：
- 请求体：message（string min 1 max 2000 trim 后非空）/ category（general | article | bug | feature，默认 general）/ articleSlug（仅 category=article 时必填，匹配 ^[a-z0-9-]+$ 且校验 slug 存在于 ArticleLoader）/ contact（string max 120 可选，如提供做基本 email 格式校验不强制）
- 响应 200：{ ok: true, id: "fb-<timestamp>-<rand6>" }
- 响应 400：{ ok: false, error: "<reason>", field: "<fieldName>" }
- 响应 429：{ ok: false, error: "rate limited" } + Retry-After header
- 限流：per IP 10 次/分钟 + 60 次/小时，内存 Map 滑动窗口，重启重置

GET /api/feedback — 列出反馈（需 auth token）：
- 请求：Authorization: Bearer <token>，query 参数 limit（1-100 默认 50）/ offset（≥0 默认 0）/ category（可选过滤）/ status（open | resolved | archived，默认 open）
- 鉴权：复用现有 authToken 机制（APTBOT_AUTH_TOKEN env var），不匹配 401
- 响应 200：{ ok: true, items: [FeedbackEntry], total: N }
- 响应 401：{ ok: false, error: "unauthorized" }

POST /api/feedback/:id/moderate — 审核/状态变更（需 auth token）：
- 请求：Authorization: Bearer <token>，body { status: resolved | archived, note?: string max 500 }
- 响应 200：{ ok: true, entry: FeedbackEntry }
- 响应 401 / 404（id 不存在）

### 存储模型

FeedbackEntry 结构：
- id（fb-<timestamp>-<rand6>）
- message / category / articleSlug? / contact?
- ip（限流 + 审计）/ userAgent?
- status（open | resolved | archived）
- note?
- createdAt（ISO 8601）
- moderatedAt?（ISO 8601）

存储格式：data/feedback.jsonl（append-only JSONL，复用现有 jsonl.ts 模式）。

为什么 JSONL 而非 SQLite：反馈量预期低（个人项目月级 <100 条）/ JSONL 与项目现有 session 存储模式一致 / 无新依赖 / 可读性好。

并发安全：复用现有 jsonl-mutex.ts per-file async-mutex（5s 超时），append 独占锁，list 读取无需锁（增量流式解析容忍并发 append）。

损坏修复：复用现有 JSONL 增量流式解析 + 破损行容错（skip + warning，不 crash）。不引入 fs.truncateSync 自动修复（反馈数据丢失可接受）。

moderate 实现：重写整个 JSONL 文件（per-file 锁保护）。考虑反馈量低，重写成本可接受，单文件原子性更好，无 tombstone 解析复杂度。

FeedbackStorage 接口：
- append(input: FeedbackInput): Promise<FeedbackEntry> — 追加一条，自动生成 id + createdAt
- list(filter: FeedbackListFilter): Promise<{ items: FeedbackEntry[]; total: number }> — 分页 + 过滤
- moderate(id: string, update: { status; note? }): Promise<FeedbackEntry | null> — 重写文件更新状态
- findById(id: string): Promise<FeedbackEntry | null> — moderate 用

### CLI /feedback 命令

注册在 CommandRegistry + CLI 实现：
- /feedback — 列出最近 10 条 open 状态（id / category / message 前 40 字符 / createdAt，Ink 表格输出）
- /feedback all — 列出全部状态（含 resolved/archived）
- /feedback <id> — 查看单条详情（含 note / moderatedAt）

### 反馈区前端交互

文章页反馈表单：method POST action /api/feedback，含 category=article 隐藏域 + articleSlug 隐藏域 + textarea maxlength 2000 required + contact maxlength 120 可选 + 提交按钮 pill。JS 拦截 submit，fetch POST，状态提示。

## ArticleLoader 实现细节与错误处理

### 加载流程

1. 扫描 src/learn/articles/*.md（fs.readdirSync 按文件名排序）
2. 逐文件 gray-matter 解析 → { data: frontmatter, content: markdownBody }。失败：stderr warning + 跳过
3. zod 校验 frontmatter（ArticleMetaSchema.safeParse 拿全部错误）。失败：stderr warning（含校验错误详情）+ 跳过
4. 唯一性校验：slug 重复（warning + 保留 order 较小者）/ prerequisites 引用不存在 slug（warning + 清空字段，不阻塞）/ track 不在 TRACKS 注册表（warning + 跳过该文件）/ order 全局重复（warning + 文件名兜底排序）
5. marked 渲染（仅 status=published）：marked.parse(markdownBody) → htmlString，缓存到内存 Map。status=planned 跳过渲染，htmlString=null
6. 排序与分组：全局按 order 升序 → 按 track 分组 → 按 chapter 分组 → 按 order 升序
7. 返回 ArticleState（不可变快照，含 articles + tracks + bySlug Map + byTrack Map）

### 热重载策略

与 v0.2.2 Config 热重载模式对齐：mtimeNs 懒加载，非 fs.watch。

触发时机：每次 HTTP 请求 /learn 或 /learn/:slug 时，ArticleLoader 检查 articles/ 目录 mtimeNs + 每个文件 mtimeNs 数组快照。任一变化触发 reload。

性能：目录 + 文件 mtimeNs 检查约 5ms（19 文件 stat），重新扫描约 50ms。高并发下用 per-loader mutex 串行化 reload，并发请求等同一个 reload 完成。

与 Config 热重载独立工作（不联动）。

### marked 渲染配置

marked.setOptions gfm true / breaks false。

自定义 renderer 为 h2/h3 生成 id（slugify：转小写 → 非字母数字替换为 - → 连续 - 压缩 → 去首尾 -，冲突追加 -2/-3）。为 pre 加 data-language 属性。

marked@15 renderer API 为 token-based（renderer.heading({ tokens, depth })），具体 API 以安装版本为准，TDD 驱动适配。

### XSS 防护

文章源文件由开发者维护（信任边界内），不引入 DOMPurify。marked 默认转义内联 HTML 中的 <。测试覆盖"恶意 HTML 输入应被 marked 转义"。锁定 marked major 版本（^15），package-lock.json 固定具体版本。

不引入语法高亮（marked 输出纯 pre/code，用 CSS 等宽字体 + bg-darker 背景区分）。

### 边界情况

- articles/ 目录不存在或为空：返回空 ArticleState + stderr warning，/learn 路由显示"暂无文章"占位（不返回 404），/learn/:slug 全部 404
- 全部文章 status=planned：/learn 列表页显示全部卡片（半透明 + coming soon），/learn/:slug 显示 coming soon 占位
- published 但 markdownBody 为空（trim 后 < 100）：stderr warning（疑似未写完），不阻塞，仍渲染
- order 全局重复：warning + 文件名兜底排序
- 热重载期间请求到达：per-loader mutex 串行化，不存在读到半旧半新问题
- 文件编码：强制 UTF-8，gray-matter 自动剥离 BOM

### 风险与缓解

1. marked 版本 API 变更：marked@15 的 renderer API 是 token-based，与旧版不同。缓解：Task 1.2 实施时先查 marked@15 文档确认 API，TDD 驱动适配。
2. 19 篇文章撰写周期：每篇 1500-2500 字，总 3-5 万字，是 0.2.3 最大工作量。缓解：Task 6 可与组 1-5 并行；如时间紧可降级部分为 planned。
3. gray-matter ESM 兼容性：gray-matter@4 是 CommonJS，aptbot 是 ESM 项目。缓解：用 import matter from 'gray-matter'（Node.js ESM 互操作支持 default import）；如不兼容改用 js-yaml 手动解析 frontmatter（成本约 30 行代码）。
4. 反馈区 spam 风险：VPS 公网暴露 POST /api/feedback。缓解：per IP 限流（10/min + 60/hour）+ message 长度限制；未来如 spam 严重可加 CAPTCHA 或 IP 黑名单。
5. 文章 markdown 中的 HTML 注入：marked 不剥离 raw HTML。缓解：文章由开发者维护（信任边界内）+ 测试覆盖恶意 HTML 输入应被 marked 转义。
