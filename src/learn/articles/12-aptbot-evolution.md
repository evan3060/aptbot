# aptbot 开发全过程：MVP → 0.2.2 演进回顾

前面 11 篇文章讲的是 aptbot 的"是什么"——各个子系统如何设计。这篇文章讲"怎么来的"——从 MVP v0.1.0 到 v0.2.2 的演进过程，每个版本的核心决策、引入的抽象、解决的问题。这是"项目即学习"理念的具体展开：演进路径本身就是教材。

## MVP v0.1.0：42 任务 / 54 源文件 / 383 测试

aptbot 的 MVP 在 42 个任务里完成，54 个源文件，383 个测试。这是个什么规模？对比一下：一个"教程级"的 agent demo 可能就 5-10 个文件，aptbot MVP 是它的 5 倍多。

MVP 阶段建立的核心结构：

- **四层架构**（access/bus/core/infrastructure + shared）从第一天就立起来。这不是后加的，是初始设计。
- **AgentLoop + Provider + ToolRegistry + Memory** 的核心循环跑通。能接收消息、调 LLM、执行工具、记历史、返回响应。
- **CLI（Ink）+ WebSocket 服务端**双前端接入。CLI 是开发主用，WebSocket 服务端是为后续 WebUI 准备。
- **JSONL 持久化**选型确定。session、memory 都用 JSONL，复用 jsonl.ts 与 jsonl-mutex.ts 工具。
- **测试基线 383**：每个核心模块都有单元测试，集成测试覆盖关键路径。

MVP 的核心决策："先打通垂直切片"。不是把 Provider 做完美再上 Tool，而是 Provider 够用就上 Tool，Tool 够用就上 Memory，每个子系统"够用"就推进。这让 42 任务能覆盖完整 agent 闭环——如果一个子系统打磨到完美才进下一个，MVP 可能要 100+ 任务。

MVP 解决的问题："agent 能跑起来"。agent 能接收用户消息、调 LLM、用 bash/read/edit/update_working_memory 四个工具做事、把结果回给用户。这听起来朴素，但是 agent 的本质——后面的所有工作都是在这个核心循环上加可靠性、加能力、加 UX。

## v0.2.0 L1：用户系统 + 多客户端同步 + 侧边栏

v0.2.0 是 aptbot 第一次大版本演进，引入三个相互关联的能力：

**用户系统**：UserStorage + scrypt 密码哈希 + Bearer token 鉴权。这是多用户隔离的基础。没有用户系统，所有 session 都属于"匿名用户"，谁都能看谁的，多用户场景完全不可用。

**多客户端同步**：Channel 抽象 + bindSession 多对一 + presence + session_changed。这是"agent 状态独立于客户端"的实现。引入后，用户能在电脑、手机、平板之间无缝切换，agent 状态不丢。

**侧边栏**：WebUI 加 session 列表侧边栏，用户能管理多个 session、切换、删除、标签。这是多 session 场景的 UX 基础——没有侧边栏，用户只能通过 CLI 命令管理 session，门槛太高。

这三个能力是耦合的——用户系统支撑多客户端同步（识别谁是谁），多客户端同步支撑侧边栏（多个 session 在多端可见）。v0.2.0 把它们一起做，而不是分三个小版本，因为单独做任一个价值都有限。

v0.2.0 引入的抽象：

- **Channel 接口**：多端接入的基础抽象
- **bus + AgentEventEnvelope**：事件流的标准封装
- **UserStorage 接口**：用户存储的抽象（未来可换 SQLite、PostgreSQL）
- **SessionRef**：可变引用，支撑运行中切 session

测试基线 383 → 584，新增约 200 项测试。这个增量反映了"用户系统 + 多客户端同步"的复杂度——鉴权、session ownership、presence、resync 协议每个都需要测试覆盖。

## v0.2.1：aptbot.de 落地页 + adept.ai 风格克隆

v0.2.1 是个 UX 重的版本，做"对外门面"——aptbot.de 落地页。

**落地页**：5 个 section（hero / features / architecture / use cases / CTA），用 adept.ai 的视觉风格（白底 + 深绿 + Inter 字体 + pill 按钮）。这不是功能演进，是"项目对外形象"的演进——aptbot 不再只是"自己用的工具"，开始有"展示给别人看"的形态。

**adept.ai 风格克隆**：选 adept 作为视觉参考，是因为 adept 也是 agent 产品，视觉风格契合 agent 调性。克隆不是抄设计稿，是学视觉 tokens（颜色、字号、间距、按钮样式），用这些 tokens 渲染 aptbot 自己的内容。

**Hero 副标题 + 数据条**：落地页 hero 区域加副标题描述 aptbot 定位，加数据条显示测试数 / 文件数 / 模块数等"项目健康度"指标。这让落地页不只是"营销页"，也是"项目仪表盘"。

**Demo 模式**：落地页加 demo 入口，访客能直接在网页上试用 aptbot，不需要自部署。这是"降低体验门槛"的演进——让人 30 秒内感受到 aptbot 能做什么。

v0.2.1 的核心决策："UX 也是能力"。一个能跑但难看的 agent，不如一个能跑且好看的 agent 有价值。视觉、文案、交互流程都是 aptbot 的"能力"，不是附属品。

测试基线 584 → 687，新增约 100 项测试。增量主要是 HTML 生成器的断言测试——落地页是字符串生成的 HTML，测试用"含/不含特定字符串"断言，沿用 v0.2.1 模式（不需要拉起浏览器）。

## v0.2.2：可靠性 + 可扩展性 + UX（10 项核心能力）

v0.2.2 是 aptbot 最大的一次演进，10 项核心能力一起做：

1. **Provider 错误分类重试**：401/403/400 fatal，429/5xx 退避重试，ECONNRESET 网络重试
2. **MixinProvider 故障转移**：多 provider 优先级 + springBackMs 弹回
3. **双时钟流式控制**：TTFB 5s + 块间 1.5s
4. **流式后不切 provider**：保住正确性优先于完整性
5. **Tool 30s 硬超时**：SIGTERM→SIGKILL 两阶段
6. **大文件 OOM 防护**：read 工具文件大小检查
7. **path-guard 路径遍历防护**：workspace 内绝对路径校验
8. **JSONL 损坏自修复**：fs.truncateSync 截断到完整行
9. **Hook 系统 8 扩展点**：agent/turn/llm/tool × before/after
10. **Skills 系统**：两层加载 + L1 索引 + 热重载

这 10 项能力可以分成三类：

**可靠性**（1-4）：Provider 层的故障转移与流式控制。让 agent 在 provider 不稳定时不卡死、不丢数据、能自动恢复。

**安全**（5-8）：Tool 层的超时、OOM、路径防护、JSONL 修复。让 agent 不会因为自己的"探索"行为卡死或损坏数据。

**可扩展性**（9-10）：Hook 与 Skills。让 aptbot 不需要改源码就能扩展行为与知识。

v0.2.2 的核心决策："可靠性优先于新功能"。0.2.2 没有加新的"酷能力"（如新工具、新 provider 协议），全部工作都在"让现有能力更可靠"。这是项目从"能用"走向"能长期用"的关键转折——MVP 与 0.2.0/0.2.1 加能力，0.2.2 加韧性。

测试基线 687 → 938，新增约 250 项测试。这个增量是四个版本里最大的，反映了"可靠性"工作的测试密度——每类错误、每个重试路径、每个故障转移场景都需要测试覆盖。可靠性不能"看起来工作"，必须有测试证明它在各种边界条件下工作。

## 每个版本的核心决策、引入的抽象、解决的问题

| 版本 | 核心决策 | 引入的抽象 | 解决的问题 |
|---|---|---|---|
| MVP | 先打通垂直切片 | 四层架构 + AgentLoop + Provider + ToolRegistry + Memory | agent 能跑 |
| 0.2.0 | 用户/多端/侧边栏一起做 | Channel + bus + UserStorage + SessionRef | 多用户多端可用 |
| 0.2.1 | UX 也是能力 | LandingPage HTML 生成器 + 视觉 tokens | 对外门面 |
| 0.2.2 | 可靠性优先于新功能 | MixinProvider + Hook + Skills + path-guard | 长期可靠运行 |

每个版本的"核心决策"都是一次取舍：MVP 选"完整闭环 > 完美子系统"，0.2.0 选"耦合能力一起做 > 渐进加分",0.2.1 选"投入 UX > 加功能"，0.2.2 选"加韧性 > 加能力"。这些决策不是孤立的，是层层递进——MVP 跑起来才能做多端，多端做出来才能展示，展示完才能投入韧性。

## 测试基线演进：383 → 584 → 687 → 938

测试基线数字本身就是 aptbot 演进的"客观记录"：

- **MVP 383**：核心循环测试 + 基础工具测试 + JSONL 持久化测试
- **0.2.0 584（+201）**：用户系统 + 多客户端同步 + presence + resync 协议测试
- **0.2.1 687（+103）**：HTML 生成器断言 + 落地页移动端测试
- **0.2.2 938（+251）**：Provider 故障转移 + 错误分类 + Hook + Skills + 安全防护测试

每个版本的测试增量反映了"新增复杂度的可测试部分"。0.2.2 增量最大（+251），因为可靠性工作的测试密度最高——每条错误路径、每个故障转移场景都必须有测试。

测试基线还有个隐性约束："不退化"——后续版本不能让既有测试 fail。这强制每个版本演进时不破坏前面的能力，是"加法而不减法"原则的客观保证。

## 架构演进的"加法而不减法"原则

aptbot 演进遵循"加法而不减法"——新版本加抽象、加能力、加测试，但不删旧抽象、不删旧能力、不删旧测试。

为什么这条原则重要？

1. **降低回归风险**：删一个旧抽象可能让依赖它的代码崩，加新抽象不会。删是高风险操作，加是低风险操作。
2. **保留演进痕迹**：旧抽象是新抽象的"前身"，保留它能让人理解"为什么有新抽象"——新抽象解决了旧抽象的什么问题。
3. **支持渐进迁移**：新抽象上线后，旧抽象可以继续工作，用户按自己节奏迁移。不需要"一次性切换"。
4. **降低认知成本**：用户已经学会的抽象还在，新抽象是"额外学一点"，不是"重新学一遍"。

这条原则的代价是"抽象越来越多"——aptbot 0.2.2 比 MVP 多了不少抽象（Channel、bus、MixinProvider、Hook、Skills…），代码量也在增长。但每个抽象都有清晰的存在理由，不是"为了加而加"——它们各自解决一个具体问题。

"加法而不减法"不意味着永远不重构。当某个抽象确实过时（如被新抽象完全取代），aptbot 会做删除——但这是有意识的"封仓重构"，不是顺手删。每次删除都伴随着测试更新与文档同步，确保删得安全。

## 小结

aptbot 从 MVP 到 0.2.2 的演进路径，是"项目即学习"理念的具体展开。MVP 立核心循环，0.2.0 加多用户多端，0.2.1 投入 UX，0.2.2 加可靠性。每个版本的核心决策都是一次明确取舍，引入的抽象都解决具体问题，测试基线 383→584→687→938 客观记录了演进复杂度。"加法而不减法"原则保证了演进不破坏既有能力，让项目能长期累积而非反复重写。
