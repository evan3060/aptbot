# spec 文档管理：从 brainstorming 到归档的全生命周期

AI 辅助开发里，spec 是被低估的资产。很多人觉得"写 spec 比写代码还慢，不如直接让 AI 写"。但缺少 spec 的项目，每个迭代都要重新跟 AI 解释一遍"为什么这么设计""什么不能改""什么是未来版本"，重复成本远超一次 spec 撰写。本篇梳理 spec 从诞生到归档的全生命周期。

## 为什么需要文档化的 spec

AI 没有"项目记忆"。每次会话开始，它对项目的理解全靠当前上下文里的文档。如果没有 spec，AI 会基于"通用最佳实践"做决策——引入它熟悉的抽象、按它习惯的模式组织代码、把它认为"应该有"的功能加进来。这些决策之间相互冲突，几次迭代后项目就变成四不像。

spec 的核心价值是**固化决策**。把"为什么选 A 不选 B""什么是 in scope 什么是 out of scope""哪些是硬约束"写下来，AI 每次会话读到 spec，决策就一致。三个月后你自己回看，也能立刻回忆起当时的判断依据，不用从代码反推意图。

文档化的 spec 还有一个隐含价值：**写出来才发现没想清楚**。脑子里觉得"这个功能就这么做"，落到 spec 里要写"输入是什么、输出是什么、错误怎么处理、与现有模块怎么交互"，写着写着就发现某个分支根本没想过。spec 是把模糊想法逼成清晰决策的工具。

## spec 与 plan 与 design 的边界

三类文档容易混淆，混淆会导致 AI 越界。边界划分：

- **spec**：描述性 + 决策性。回答"做什么、为什么这么做、不做什么"。范围、目标、验收标准、开放问题决策表。**不含代码、不含实现步骤**。
- **plan**：执行性。回答"按什么顺序做、每步怎么验证"。subtask 清单 + 验证命令。**不含代码、不含设计理由**。
- **design**：技术性。回答"用什么技术方案实现"。架构图、接口签名、数据结构、关键算法。**是 spec 与 plan 之间的桥梁**。

常见的越界：spec 里写实现代码（应该挪到 design 或实现阶段）、plan 里写设计理由（应该回 spec）、design 里写任务清单（应该挪到 plan）。一旦越界，文档职责模糊，AI 不确定某条信息是"约束"还是"建议"，执行时就会打折扣。

简单判断法：拿着一段内容问"这是约束、是步骤、还是技术？"——约束归 spec，步骤归 plan，技术归 design。

## spec 文件组织结构

文件命名约定：`docs/superpowers/specs/YYYY-MM-DD-<topic>-design.md`。

- `YYYY-MM-DD` 是 spec 创建日期，便于按时间排序与回溯。
- `<topic>` 是主题短词，比如 `0.2.3-learn-system`、`auth-redesign`、`api-rate-limiting`。
- `-design.md` 后缀表明这是设计文档，与 plan（`-plan.md`）、dev-workflow（`-workflow.md`）、design-notes（`-notes.md`）区分。

所有 spec 集中在 `docs/superpowers/specs/` 目录，不分散在模块目录。原因有二：一是便于全局检索——"这个项目做过哪些设计决策"一眼可见；二是便于跨模块引用——A 模块的 spec 可能引用 B 模块的 spec，集中存放路径稳定。

目录下文件多了之后（几十份 spec），可以按年份或版本分子目录，但不要按模块分——按模块分会破坏时间线，难以回溯"某段时间这个项目在做什么"。

## spec 的生命周期

一份 spec 不是写完就结束，它有完整生命周期：

1. **brainstorming 产物**：brainstorming 阶段的决策表，是 spec 的初稿来源。决策表里每条开放问题的"决策 + 依据"直接搬进 spec 对应章节。
2. **user review**：spec 写完 + self-review 通过后，提交用户审核。用户提出修改、补充、否决。这一步 spec 是"待批"状态。
3. **实施依据**：user review 通过后，spec 成为后续 plan 与实现的"宪法"。任何实现决策与 spec 冲突，要么改实现，要么改 spec，不能"私下偏离"。
4. **归档**：版本发布后，spec 归档为该版本的设计记录。归档不是删除，是标记为"历史版本的设计依据"，未来回溯时能查到。

生命周期的每个阶段 spec 状态不同：草稿 → 待批 → 生效 → 归档。状态可以写在 spec 文档头部（"状态：待用户审核"），让读者立刻知道这份 spec 是否还生效。

## spec self-review 检查清单

self-review 是 spec 提交用户审核前的自检。检查清单：

- **placeholder 残留**：搜索 `TODO`、`TBD`、`待定`、`xxx`、`???`，所有 placeholder 必须填实或删除。
- **前后一致性**：前文说"支持 5 种协议"，后文列表只有 4 种；前文说"默认 true"，配置示例里写 `false`。这类矛盾 self-review 时逐条对照。
- **范围合理**：in scope 与 out of scope 划分清晰，没有"既算 in 又算 out"的模糊项。每个 in scope 项有对应的验收标准。
- **歧义表述**：搜"大致""可能""看情况""视具体情况""灵活处理"——这些词都是歧义信号，要么具体化，要么明确为开放问题。
- **决策依据**：每条决策后面有"为什么"。没有依据的决策等于没决策，因为下次会有人（或 AI）质疑。

self-review 不是走形式，要真的逐项过。AI 自己 review 自己写的 spec 有盲区，可以换一个 agent 角色做 review（"你现在扮演挑剔的架构师，找出这份 spec 的问题"），视角转换能暴露更多问题。

## spec 与代码的同步

spec 不是写完就冻结。实现过程中会发现 spec 的某些决策不可行，需要调整。同步规则：

- **spec 变更必须留痕**。在 spec 里加一个"变更记录"章节，记录"日期 + 变更点 + 原因"。不要直接覆盖原文，保留决策演进。
- **变更触发 user review**。重大变更（范围调整、架构变更）要重新过 user review，不能 AI 自己拍板改 spec。
- **过时 spec 标记**。如果某条 spec 决策被后续版本推翻，在 spec 里标注"v0.x.y 已变更，参见 YYYY-spec"，不要让读者误以为还生效。

spec 与代码同步的典型问题：代码改了 spec 没改、spec 改了代码没改、两者都改了但方向不一致。封仓时要做一次 spec 与代码的对账，确认 spec 描述的行为与代码实际行为一致。对账发现的差异，要么改代码要么改 spec，不能留"已知差异"。

## 多版本 spec 的归档与回溯

每个版本一份 spec，归档后形成历史序列。归档的价值：

- **决策回溯**：半年后问"为什么当时选了 A 不选 B"，查当时的 spec 就知道，不用猜。
- **责任追溯**：某条决策导致问题，查 spec 能看到决策依据，判断是"依据错了"还是"执行偏离了"。
- **新人 onboarding**：新加入的 AI 会话或人类协作者，读历史 spec 能快速理解项目演进逻辑。

归档用 git history + specs/ 目录双重保险。git history 保留所有版本变更，specs/ 目录保留每份 spec 的最终态。两者配合：想知道"v0.2.0 时这个模块怎么设计的"，查 specs/ 目录里 v0.2.0 日期的 spec；想知道"这个决策是哪次 commit 改的"，查 git log。

## spec 作为协作媒介

spec 不只是写给 AI 的，也是团队协作的媒介：

- **PR review 依据**：reviewer 看 PR 时，对照 spec 判断"实现是否符合设计"。PR 偏离 spec 要么改 PR 要么改 spec。
- **新人 onboarding 文档**：新人入项目，先读最新版本的 spec，再读代码。spec 是"设计意图"，代码是"设计落地"，先读意图再看落地，理解更快。
- **跨团队对齐**：多个团队协作时，spec 是接口契约。各团队的实现都对照同一份 spec，避免"我以为你以为"。

即使是个人项目，把 AI 当作"协作者"，spec 也是你与 AI 之间的契约。每次会话开始把 spec 喂给 AI，比每次重新解释省心得多。

## 何时该写 spec、何时不必

不是所有改动都需要 spec。判断标准：

**该写 spec**：系统性变更（新增模块、重构架构、引入依赖、调整核心抽象）、跨版本规划（路线图、版本规划）、涉及多个模块协同的功能、有重要决策需要记录的改动。

**不必写 spec**：trivial 改动（改个文案、修个 typo、调整常量）、单点 bug 修复（修一个明确的小 bug）、纯实验性探索（还没决定要不要做的试验）。

判断尺度："这个改动会影响哪些模块？""半年后还需要知道为什么这么改吗？"——影响多模块、需要长期记忆的，写 spec；局部、一次性的，直接改。

过度写 spec 与不写 spec 一样有害。给每个 typo 都写 spec，spec 价值被稀释，重要的设计文档淹没在噪音里。spec 要写在该写的地方，让它成为"重要决策的索引"而非"所有改动的流水账"。

## 小结

spec 是 AI 辅助开发的"项目记忆"与"决策宪法"。从 brainstorming 诞生，经 self-review 与 user review 把关，成为实施依据，最终归档为历史记录。与 plan 和 design 边界清晰，与代码同步演进，多版本可回溯。写 spec 的成本是一次性的，不写 spec 的成本是每次迭代都要重新解释——长期算下来，spec 是省时间的投资。
