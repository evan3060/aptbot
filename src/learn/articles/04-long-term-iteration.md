# 长期迭代维护实践：架构演进与可持续性

让 AI 写出能跑的代码不难，难的是让项目在 5 个、10 个版本之后还能继续演进。很多 AI 辅助项目死于"第三个版本"——前两个版本快速堆功能，第三个版本开始发现改一处崩三处、加功能没地方放、AI 每次会话都建议推翻重来。本篇梳理长期迭代维护的实践，核心是**让架构随版本加法演进，让债务随版本可控偿还**。

## 版本规划：L1/L2/L3 路线图 vs 单版本 spec

长期项目需要两层规划：

- **路线图（L1/L2/L3）**：粗粒度的长期方向。L1 是当前版本要做的，L2 是下两三个版本的方向，L3 是远期愿景。路线图不细到任务，只到大方向——"L2 会引入 IM 集成""L3 探索自演化能力"。
- **单版本 spec**：细粒度的当前版本设计。只覆盖 L1，详细到文件、接口、测试用例。

两者的关系：路线图给 spec 提供方向，spec 给路线图提供落地。spec 阶段做决策时，要对照路线图判断"这个决策是否会堵死 L2 的路"。比如 L2 计划引入 IM 集成，L1 的 spec 在设计 Channel 抽象时就要预留 IM 接入点，不能把 Channel 写死成 WebSocket-only。

路线图不是合同，是方向。L2 真到了那个版本，可能因为技术变化、需求变化调整方向，这是正常的。但调整要有意识——明确"我们改变了路线图"，而不是"忘了原来计划做什么所以随便来"。路线图写在 design-notes 里，定期回看、定期校准。

## 迭代节奏：封仓 → UAT → 下一版本

迭代不是连续滚动的，是节奏化推进。每个版本一个完整循环：

1. **规划期**：写 brainstorming + spec + plan。这个阶段不写代码，只设计。
2. **实现期**：按 plan 逐 subtask 执行，TDD 红→绿。
3. **封仓期**：finishing-a-development-branch 收尾，文档同步，tag。
4. **UAT 期**：四类核验清单逐项过。
5. **发布**：合并 main、推 tag、部署。
6. **冷却期**：发布后短暂冷却，不立即开下一版本。整理 design-notes、回顾本版本踩的坑、调整工作流。

节奏化比连续滚动好，因为它给"反思"留出位置。连续滚动开发容易陷入"加功能→出 bug→修 bug→加功能"的死循环，没有抬头看路的时间。冷却期是抬头的时间——回看本版本哪些 spec 决策事后看错了、哪些 AI 行为模式需要新约束、哪些技术债该排进下个版本。

## 测试基线管理：不退化红线

长期迭代里测试基线是最敏感的健康指标。每个版本封仓时记录"测试总数 / 通过数"，下一版本对照：

- **总数只增不减**：新功能配新测试，总数应该上升。总数下降要么是删了测试（要追问为什么），要么是 skip 了（要追问为什么）。
- **通过率不退化**：上个版本 936/938（2 flaky），这版本不能变成 920/950。通过率下降意味着引入了新 bug 或新 flaky。
- **flaky 必须治理**：长期项目里 flaky 测试是慢性毒药。一开始一两个偶发失败，几个月后变成"测试红了正常，等等就绿"，红灯失去警示意义。每个版本必须把 flaky 数量压到零或可解释的极少数。

历史遗留 flaky 的处理：能修则修（通常是时序、并发、外部依赖问题），不能修的隔离到单独的 test suite 不计入主基线，不能让它们混在主测试里污染信号。

## 架构演进：加法而不减法

长期项目的架构演进遵循"加法而不减法"原则：新版本叠加新抽象，不删除老抽象（除非有明确的重构版本）。

为什么？因为删除抽象会破坏向后兼容，而 AI 辅助项目里"谁在用这个抽象"经常不清楚——AI 写的代码可能引用了某个内部 API，人类记不住所有引用点。贸然删除，下游某处就崩了。

加法的做法：新抽象与老抽象并存，新功能用新抽象，老功能维持老抽象，逐步迁移。比如 v0.1 用 plain HTTP，v0.2 引入 WebSocket，两者并存；v0.3 引入 Channel 抽象，新接入用 Channel，老的 HTTP/WS 维持。直到某个版本专门做"统一抽象"重构，才删除老路径。

加法原则的代价是代码量膨胀、抽象层叠加。这是可控的代价——比"每次重构推翻一切"的代价小得多。膨胀到一定程度（比如三层 wrapper 嵌套），安排一个专门的重构版本收敛，不要在日常功能版本里顺手重构。

## design-notes 跨版本设计笔记

design-notes 是跨版本的设计记忆，记录约束、原则、决策回溯。与 spec 的区别：spec 是单版本的设计，design-notes 是跨版本的累积。

design-notes 该记什么：

- **硬约束**：项目里不能违反的规则。比如"core 层不能 import access 层""API key 只能从 env 读""测试不能依赖外部网络"。这些约束散落在各处 spec 里，design-notes 汇总成一份。
- **原则**：项目遵循的设计原则。比如"加法而不减法""YAGNI""错误不持久化"。
- **决策回溯**：重要决策的演进。比如"v0.1 用 plain HTTP，v0.2 加 WS，v0.3 引入 Channel 抽象，原因是 X"。这条线在单版本 spec 里看不到全貌，design-notes 串联起来。
- **教训**：踩过的坑。比如"曾经因为 AI 跳过测试导致发布后 hotfix，从此 TDD 设为红线"。

design-notes 是项目最长期价值的文档。代码会重写、spec 会归档，design-notes 持续累积。新人 onboarding 读 design-notes 比读所有 spec 更高效——它压缩了项目的"为什么"。

## project_memory 维护

project_memory 是给 AI 看的项目记忆。与 design-notes 区别：design-notes 写给人，project_memory 写给 AI，是每次会话注入 system prompt 的"项目宪法"。

project_memory 该包含：

- **hard constraints**：硬约束，AI 必须遵守。违反这些就是越界。
- **lessons learned**：踩过的坑及对应约束。比如"AI 倾向跳过测试，必须用 skill 强制 TDD"。
- **当前版本焦点**：本版本在做什么、不做什么，避免 AI 跑偏到未来版本。
- **架构地图**：分层结构 + 单向依赖规则，AI 改代码时知道往哪放。

project_memory 要精简。它会注入每次会话，太长浪费 token、稀释信号。目标是几百字到一千字，覆盖最关键的约束与原则。详细的设计放 spec 与 design-notes，project_memory 只放"AI 必须知道的"。

## 重构时机判断

重构是必要的，但时机要对。判断标准：

**该重构**：抽象层叠加到明显混乱（三层 wrapper 嵌套）、某模块改动成本远超功能价值（改一行要动五个文件）、测试基线长期 flaky 暗示架构有问题、新人 onboarding 成本陡增。

**不该重构**：功能版本里顺手重构（重构应该独立成版本）、没写测试的模块重构（重构后无法验证行为不变）、AI 建议重构（AI 经常建议"推翻重来"，因为它没有沉没成本概念，要警惕）、为了"代码好看"重构（重构要为了"可维护性"，不是为了审美）。

重构版本的特征：不增加新功能，只调整结构；测试基线总数不变（行为不变），通过率必须 100%；spec 明确"重构范围"与"不重构范围"，避免蔓延。

## 依赖升级策略

AI 辅助项目里依赖升级要克制。每引入一个新依赖都要评估：

- **必要性**：能不能用标准库或现有依赖解决？YAGNI 原则优先。
- **维护活跃度**：last commit、release frequency、issue 响应。死的依赖别引。
- **依赖体积**：是否带来大量传递依赖，是否影响 bundle size。
- **兼容性**：与现有依赖版本冲突吗？ESM/CJS 兼容吗？
- **安全**：有已知 CVE 吗？锁定 major 版本，package-lock.json 固定具体版本。

升级已有依赖也要谨慎。minor 升级通常安全，major 升级要单独安排——读 changelog、跑全量测试、UAT 核验。AI 倾向于"用最新版本"，但不最新不一定是坏事，稳定比新鲜重要。

## 文档同步：CHANGELOG / README / ARCHITECTURE

三份文档与版本同步：

- **CHANGELOG**：每个版本一个条目，分 Added/Changed/Fixed/Removed。用户视角，不技术细节。
- **README**：Features 表、快速开始、配置说明。新功能加进 Features，配置项更新到说明。
- **ARCHITECTURE**：模块说明、分层结构、设计原则。新增模块加章节，架构调整更新结构图。

封仓时这三份必须与代码同步。不同步的文档比没有文档更糟——读者按文档操作发现不对，会失去对文档的信任，从此文档变成摆设。

## 项目可持续性

长期项目的可持续性是综合指标：

- **技术债可控**：每个版本偿还一点，不让债务滚雪球。
- **代码可读**：新人（或新 AI 会话）能快速理解。命名清晰、模块边界明确、抽象不过度。
- **上手成本不陡增**：onboarding 时间应该随版本稳定或下降，不能每个版本都更难上手。
- **测试可信**：测试红了一定是真坏了，不是噪音。
- **文档与现实一致**：文档不滞后代码超过一个版本。

可持续性的反面是"加功能越来越慢"。如果 v0.1 加一个功能用 1 天，v0.5 加同样的功能用 5 天，说明项目在失去可持续性——通常是抽象混乱、测试不可信、文档过时三者的叠加。这时候该停下来做整理版本，而不是继续硬加功能。

## 小结

长期迭代维护的核心是**节奏与节制**。节奏化推进给反思留位置，加法原则给兼容留余地，design-notes 给记忆留载体，测试基线给健康留信号。可持续性不是某项技术，而是几十个版本的累积选择——每个版本选择"加一点、还一点、记一点"，项目就能长期演进；每个版本选择"快加功能、欠技术债、忘决策"，项目就在第三个版本死掉。
