# 开发流程与约束：superpower 工作流

让 AI 写代码这件事，门槛不在"能不能写"，而在"写得可控"。一个不受约束的 agent 会跳过测试、擅自在计划文档里塞实现代码、把失败方案重复试十遍。本篇梳理一套在实践中反复验证可行的标准流程——superpower 工作流——它的核心不是某个技巧，而是用文档分层与硬约束把"自由发挥"变成"可预测推进"。

## 标准流程：brainstorming → spec → plan → TDD 实现

一条完整功能从想法到上线，经过四个阶段，每个阶段产出不同文档、解决不同问题：

1. **brainstorming（头脑风暴）**：把模糊需求变成决策表。每一条开放问题列出选项 + 决策 + 依据，避免"我以为你以为"的歧义。
2. **spec（规格说明）**：把决策固化为系统设计。范围、架构、文件结构、测试策略、验收标准全部写在 spec 里，是后续一切工作的"宪法"。
3. **plan（实施计划）**：把 spec 拆成可执行任务清单。每个 subtask 一行描述 + 验证命令，按依赖排序，看板化推进。
4. **TDD 实现**：按 plan 逐项执行，红→绿循环驱动代码诞生。

四个阶段不能跳过。跳过 brainstorming 直接写 spec，会在 spec 阶段反复推翻决策；跳过 spec 直接写 plan，会在实现阶段发现"这个抽象根本不该存在"；跳过 plan 直接 TDD，会失去任务级回归点，一个 subtask 的错误蔓延到下游。

这条流程的价值不是"严谨"，而是**让 AI 的输出可追溯**。当某个实现出问题，能回溯到 plan 的哪条 subtask、spec 的哪条决策、brainstorming 的哪个选项，问题定位从"模型为什么这么写"变成"我们在哪一步放错了约束"。

## 文档分层：plan / design / dev-workflow / design-notes

四类文档各司其职，混淆会直接导致 AI 越界：

- **plan**：描述性文档，只说"做什么 + 怎么验证"，不含代码。
- **design**：技术方案文档，含架构图、接口签名、关键算法，是 plan 的技术依据。
- **dev-workflow**：工作流文档，定义流程约束（比如"每个 subtask 完成必须跑 tsc"），是过程规范。
- **design-notes**：跨版本设计笔记，记录约束、原则、决策回溯，是长期记忆。

最常被违反的边界是 plan 里写实现代码。一旦 plan 出现代码，AI 会把 plan 当成"待粘贴的草稿"，跳过 TDD 直接复制，红→绿循环失效。plan 里允许出现的"代码"只有两类：TDD 命令描述（"运行 `npx vitest run xxx.spec.ts`，预期 RED"）和验证命令，其余一律挪到 design 或实现阶段。

## plan 不含代码原则

这条原则值得单独强调。plan 的本质是**任务清单 + 验收标准**，不是代码草稿。原因有三：

第一，plan 阶段的代码无法验证。没有测试驱动，代码的正确性全凭脑补，而 AI 脑补的代码经常引用不存在的 API 或忽略边界条件。让这种代码进入仓库等于埋雷。

第二，plan 里的代码会固化 AI 的实现思路。TDD 的精髓是"先看 RED 错误信息，再写最小修复"，错误信息会引导实现方向。如果 plan 已经写好实现，AI 会无视错误信息强行套用， GREEN 阶段变成"调整测试以适配预设代码"，本末倒置。

第三，plan 里的代码会让看板失真。subtask 的"完成"应该是"测试通过 + tsc 无错"，而不是"代码已粘贴"。一旦后者成为完成标准，质量门就形同虚设。

## spec self-review + user review gate 双重审核

spec 是后续所有工作的源头，错一处波及全局。双重审核机制：

- **self-review**：spec 写完后，AI 自己按检查清单过一遍——placeholder 是否残留、前后是否一致、范围是否合理、是否存在歧义表述。这一步能挡掉大部分低级错误。
- **user review gate**：self-review 通过后，spec 提交用户审核。用户没点头之前，不允许进入 plan 阶段。

这一道 gate 不是形式主义。AI 写 spec 时经常出现的典型问题：把"未来版本"的内容写进当前 spec、把 out-of-scope 当 in-scope、用模糊词（"大致""可能""看情况"）掩盖未决策点。user review 是把这些"软错误"逼出来的最后一道关。

## subagent-driven-development 自动推进任务链

plan 拆成 subtask 后，subagent-driven-development 模式让任务链自动推进：主 agent 把每个 subtask 委派给子 agent，子 agent 完成 + 自验 + 提交后，主 agent 自动派发下一个。

这套机制的关键在于**子 agent 的隔离性**。每个子 agent 只看到自己 subtask 的上下文，不被全局噪音干扰；完成后返回结构化报告（做了什么 / 改了哪些文件 / 测试结果），主 agent 据此决策是否继续。这种"主控 + 子执行"的分层，比单 agent 串行做完所有任务更稳定，因为单 agent 上下文越长越容易跑偏。

但自动化不是全自动。每个 subtask 之间应有一个"见证点"——主 agent 必须确认子 agent 的测试真的全绿、tsc 真的没错，才能继续。跳过见证点等于把质量门交给运气。

## TDD 红→绿循环

TDD 在 AI 辅助开发里不是"最佳实践建议"，而是**唯一允许的编码方式**。循环结构：

1. **RED**：先写测试，运行，**必须在终端看到失败**。看不到 RED 就写实现，等于没写测试。
2. **GREEN**：写最小代码让测试通过。不许多写一行"顺便"的代码。
3. **REFACTOR**：测试通过后才能重构，重构后再跑一次测试确认仍绿。

为什么必须先见证 RED？因为 AI 经常写"看起来对但其实永远通过"的测试——比如断言写错变量名、测试根本没调用被测函数、mock 配置让任何输入都通过。先跑一次看到 RED，证明测试真的在测你想测的东西，GREEN 阶段才有意义。

GREEN 阶段的"最小代码"同样关键。AI 倾向于"一步到位"写出完整实现，包括各种未要求的扩展。这违反 YAGNI，也让后续 subtask 无活可干。最小代码 = 让当前测试通过的最少代码，多一行都算越界。

## 熔断机制：3 次失败止损

AI 重复尝试失败方案是高发问题。同一个测试连续失败 3 次，必须熔断：停止当前 subtask、记录失败原因、跳过、回顾。

熔断不是放弃，是**止损**。3 次失败通常意味着 AI 进入了"换种方式重试"的死循环——改了个变量名、调了个参数、调整了 import 顺序，但根本思路没变。继续试下去只是消耗 token 和时间。熔断后回顾环节要问的不是"下次怎么试"，而是"是不是这个 subtask 本身拆错了""是不是 spec 的某条决策有问题"。

熔断记录要保留。一段时间后回看熔断记录，能发现 AI 的系统性盲区——比如总是误判某个 API 的签名、总是忽略某种边界条件。这些模式是后续优化提示词与约束规则的依据。

## 看板管理：subtask 完成标记

plan 是一份 markdown 清单，每个 subtask 前面一个 checkbox。完成的标 `[x]`，进行中的标 `[~]` 或留空。这套简陋的看板看似原始，但在 AI 辅助开发里有不可替代的作用：

- **进度可见**：主 agent 一眼看到还剩几个 subtask，决定是否需要委派并行子 agent。
- **断点续传**：会话中断后恢复，看板状态就是恢复点，不用从头回顾。
- **完成判定**：一个 subtask 标 `[x]` 的前提是测试全绿 + tsc 无错 + 报告已写，三者缺一不能标记。

看板推进的节奏是"完成一个、标记一个、立即派发下一个"，无缝衔接。避免"攒一批再标记"——攒批会丢失中间状态，一旦中断就难以恢复。

## 小结

superpower 工作流的本质是用**流程约束**对抗 AI 输出的不可预测性。四个阶段不跳步、文档分层不混淆、TDD 红→绿不偷懒、熔断机制不止损、看板管理不断点。这些约束初看繁琐，但每一条都对应着 AI 真实会犯的错误。
