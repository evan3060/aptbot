# Agent 是什么：从 chatbot 到 agent 的演进

我们日常说的"AI 助手"其实跨越了三种形态：chatbot、assistant、agent。它们看起来都能"对话"，但底层范式截然不同。理解这条演进路径，是看懂 aptbot 这类项目的起点。

## chatbot / assistant / agent 的本质差异

**chatbot** 是被动响应系统。它的核心循环是 `user → reply`：用户输入一句话，模型给出一段回答，回合结束。早期的规则机器人、RASA 时代的意图识别机器人、再到 ChatGPT 默认的对话模式，本质上都属于 chatbot——你问什么答什么，模型不主动做事。

**assistant** 在 chatbot 之上加了"能力"，比如调用计算器、查天气、搜网页。但它的调用逻辑是硬编码的：开发者预先定义"当用户问天气时调用 weather_api"，模型在固定流程中触发。assistant 已经能"做事"，但做什么、何时做，由开发者决定，模型只是触发器。

**agent** 的根本转变在于：**模型自己决定做什么**。给 agent 一个目标（"帮我修这个 bug"），它会自己拆解子任务、选择工具、判断结果是否满足、循环推进，直到目标完成或主动放弃。指令从"声明式"变成"目标式"，决策权从开发者转移到模型。

| 形态 | 输入 | 决策者 | 执行 | 终止 |
|---|---|---|---|---|
| chatbot | 一句话 | 无 | 一次回复 | 用户离开 |
| assistant | 一句话 + 触发规则 | 开发者 | 单次工具调用 | 规则结束 |
| agent | 目标 | 模型 | 多步工具循环 | 模型判断完成 |

## ReAct 循环：Reasoning + Acting 交替

agent 的核心运行模式是 **ReAct**（Reasoning + Acting）。它不是新概念，2022 年由 Yao 等人提出，但直到 LLM 推理能力够强才真正可用。

ReAct 的循环结构：

1. **Reasoning**：模型思考当前状态、下一步该做什么、为什么
2. **Acting**：模型调用一个工具（bash、read、edit、搜索…）
3. **Observation**：工具返回结果，作为新的输入进入下一轮 Reasoning
4. 重复直到模型判断"任务完成"，输出最终答案

这个循环的关键不在于"调用了工具"，而在于**模型在每一步都做推理**。它不是 `if-else` 触发器，而是"我看到 read 的输出有 X 行报错，这暗示 Y 模块有问题，下一步应该 edit 那个文件"。每一步的决策都基于前一步的真实结果，不是预设脚本。

agent 与传统软件的本质区别就在这里：传统软件是**声明式指令**——开发者写好每一步、每个分支；agent 是**自主决策**——开发者只提供工具和目标，路径由模型现场决定。这意味着同一个 agent 面对同一个任务，两次执行可能走完全不同的路径。

## agent 核心组件心智模型

把 agent 拆开看，可以归纳为四个核心组件，用人体比喻：

- **LLM 大脑**：负责推理、决策、语言理解与生成。是 agent 的"思考器官"。
- **工具双手**：让 agent 能"做事"。bash 让它执行命令、read 让它看文件、edit 让它改代码。没有工具，agent 只是会说话的 chatbot。
- **记忆长尾**：短期记忆是对话历史，长期记忆是跨会话的累积。没有记忆，agent 每次都从零开始，无法承担持续项目。
- **流式嘴巴**：agent 的输出不是一次性给出，而是边思考边输出。流式不只是 UX 优化，是 agent 与人类协作的基本节奏。

aptbot 的整个架构就是围绕这四个组件展开的：`core/` 装大脑与循环、`infrastructure/` 装工具与记忆、`access/` 装流式嘴巴与多端接入。后续每篇文章都在拆解其中一个组件。

## 为什么 agent 重要

agent 的价值不在于"对话更智能"，而在于**从"问什么答什么"到"交办任务完成"**。

传统 chatbot 场景：你问"如何删除一个 git 分支"，它告诉你命令。你自己复制粘贴执行。

agent 场景：你说"把这个仓库里所有叫 old-feature 的本地分支删了"，它自己 `git branch` 列出、自己 `git branch -D` 删除、自己验证结果，给你一个完成报告。

这个转变的工程意义是：**很多原本需要人去操作的工具链，可以变成 agent 的双手**。运维、数据分析、代码维护、文档生成，凡是"有明确工具 + 可验证结果"的领域，都是 agent 的潜在应用场景。

但 agent 不是银弹。它的可靠性强依赖于：工具设计是否清晰、模型推理是否够强、记忆是否准确、错误处理是否健全。aptbot 0.2.x 系列的每一项核心工作（Provider 故障转移、Tool 安全边界、Memory 压缩、Hook 插件机制…）都是在解决"如何让 agent 更可靠"这个问题。

## 小结

agent 不是更聪明的 chatbot，而是范式不同的系统：模型从"答话者"变成"决策者"，开发者从"写指令"变成"造工具+给目标"。ReAct 循环是它的运行心跳，工具/记忆/流式是它的肢体。理解了这个心智模型，接下来 12 篇文章里讨论的每一项设计决策，都能对号入座：它属于大脑、双手、长尾还是嘴巴？它解决的是哪一类可靠性问题？
